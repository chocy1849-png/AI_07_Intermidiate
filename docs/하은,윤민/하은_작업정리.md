## 2. RAG 파이프라인

### 파일 위치
- `rag_pipeline.py` — 검색 + 생성 통합 진입점
- `src/generation/chain.py` — LLM 응답 생성 체인
- `src/generation/prompts.py` — 프롬프트 템플릿
- `src/retrieval/hybrid_retriever.py` — BM25 검색기
- `src/db/vector_store.py` — ChromaDB 벡터 검색

### 검색 구조

```
질문 입력
  ↓
검색 모드 선택
  ├─ 기본 검색  → ChromaDB 벡터 유사도 검색
  ├─ MMR       → 다양성 보장 검색 (중복 청크 억제)
  └─ 하이브리드 → 벡터 + BM25 앙상블 (α=0.5 가중 합산)
  ↓
Reranker (점수 기반 재정렬)
  ↓
상위 10개 청크 → LLM 컨텍스트로 전달
  ↓
gpt-5 응답 생성
```

### 핵심 설계 포인트

**ChromaDB 없을 때 BM25 자동 폴백**  
ChromaDB가 비어있으면 BM25 단독으로 검색. 초기 세팅 없이도 동작 가능.

**file_name 필터 (문서 특정 검색)**  
특정 문서에 대한 질문일 때 해당 문서만 검색하도록 ChromaDB 메타데이터 필터 적용.  
파일명이 DB에 없으면 필터 없이 전체 검색으로 자동 폴백.

**프롬프트 2종 분리**

| 프롬프트 | 사용처 | 거부 응답 |
|----------|--------|-----------|
| `SYSTEM_PROMPT` | Streamlit 앱 (사용자 대상) | 가능 — "문서에서 찾을 수 없습니다" |
| `EVAL_SYSTEM_PROMPT` | 자동채점 평가 전용 | 불가 — 최선을 다해 답변 강제 |

### 실행 방법

```bash

  --문제은행 "data/(260417) PartA_RFP_AutoGrading_QBank_v3_fixed.json" \
  --모드 rag \
  --출력경로 data/eval_results
```

### 채점 로직 (6단계 순서대로 시도)

```
1. 객관식 번호 매핑
   "2번" / "2. 제한경쟁입찰" / "②" → 보기 텍스트로 변환

2. 공백 제거 후 Exact Match
   "일반 경쟁 입찰" == "일반경쟁입찰" → 정답

3. 부분 문자열 포함 체크 (정답이 4글자 이상일 때만)
   "협상에 의한 계약이 적용됩니다" → 정답 "협상에 의한 계약" 포함 → 정답
   단, 앞뒤 숫자 붙어있거나 부정어 뒤면 오답 처리

4. 숫자 정규화 비교
   "352,000,000원" == "3억 5,200만원" == "3억5천2백만원" → 정답
   "49,500천원" == "49,500,000원" → 정답

5. 날짜 정규화 비교
   "2024-07-10" == "2024.07.10" == "2024년 7월 10일" → 정답

6. 숫자 목록 포함 비교
   정답 "20, 70" → 답변에 20과 70이 모두 있으면 정답
```

### 성능 개선 이력

## 4. LLM Judge (llm_judge.py)

### 파일 위치
- `scripts/llm_judge.py`
- `data/llm_judge/judge_subset_20.json` — 20문항 축소셋
- `data/llm_judge/results/` — 평가 결과 (CSV + MD)

### 설계 목적
Exact Match로 채점이 어려운 **서술형·비교형·멀티턴·거부응답** 문항의 품질을 GPT Judge로 평가.

### 평가 구조

```
RAG 응답 생성
  ↓
GPT Judge가 4가지 지표로 1~5점 채점
  ├─ Faithfulness   : 문맥에 없는 사실을 꾸며내지 않았는가
  ├─ Completeness   : 질문의 핵심 요소를 빠뜨리지 않았는가
  ├─ Groundedness   : 답변이 검색 문맥에 직접 근거하는가
  └─ Relevancy      : 질문에 직접 답하고 있는가
  ↓
평균 4.0 이상 + Faithfulness 4 이상 → PASS
평균 3.0 미만 or Faithfulness ≤ 2 → FAIL
그 외 → REVIEW
```

### 20문항 구성

| 유형 | 문항 수 | 설명 |
|------|---------|------|
| A. single_doc | 5 | 단일 문서 서술형 |
| B. comparison | 5 | 다문서 비교 (ERP, 예산 필터링 등) |
| C. follow_up | 6 | 멀티턴 (이전 답변 기억 후 드릴다운) |
| D. rejection | 4 | 문서에 없는 정보 → 거부해야 PASS |

### 실행 결과 (전체 20문항)

```
전체:     Pass 20 / Review 0 / Fail 0
평균 점수: 4.99 / 5.00

single_doc : 5/5 PASS  avg 5.00
comparison : 5/5 PASS  avg 4.95
follow_up  : 6/6 PASS  avg 5.00
rejection  : 4/4 PASS  avg 5.00
```

**rejection 유형 실제 응답 예시 (D02)**
> 질문: "극저온시스템 운전 용역이 실제로 완료된 날짜는 언제야?"  
> RAG: "완료 날짜에 대한 정보는 찾을 수 없습니다. 수행 기간은 2025년 1월 1일 ~ 12월 31일로 명시되어 있으나, 실제 완료일은 언급되어 있지 않습니다."  
> → 계획 기간을 실제 완료일로 혼동하지 않고 정확히 거부 → PASS

| 버그 | 원인 | 수정 |
|------|------|------|
| "2. 차장"처럼 번호 포함된 answer 오채점 | map 후 모델답변은 "차장"인데 truth는 "2. 차장"으로 불일치 | grade()에서 truth도 번호 제거 |
| "M+5" → "M"으로 오매핑 | "M"이 "M+5" 부분문자열이라 먼저 선택 | 정확히 일치하는 선택지 우선 반환 |
| "2025.1.1" → "1.1"로 잘림 | `^\d+\.` 정규식이 연도 숫자도 제거 | 1~2자리 숫자 + 공백 필수로 수정 |
