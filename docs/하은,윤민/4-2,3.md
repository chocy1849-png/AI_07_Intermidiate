# 4/2~4/3 작업 정리

## 1. 작업 방식 요약

이번 주 작업은 `Two-Track` 병렬 구조로 진행한다.

- `Track A (AI/RAG)`: 전처리, 청킹, 임베딩, 검색 고도화, 프롬프트 엔지니어링, 성능 평가
- `Track B (웹/백엔드/DB)`: Streamlit UI, DB 구축, 세션 관리, 배포 환경 구성
- `통합`: Track A가 만든 RAG 함수를 Track B가 호출하는 구조로 맞춘다.

핵심 산출물은 아래와 같다.

- Track A: `rag_pipeline.py`, 평가 리포트
- Track B: `app.py`, DB 파일, 배포 환경
- 통합 문서: `interface_contract.md`

#### Track B 진행 중 함께 확인한 Track A 선행 작업

Track B를 중심으로 작업했지만, UI와 DB를 실제 문서 데이터에 연결하려면 전처리와 청킹 결과가 먼저 필요해서 아래 항목도 선행 확인했다.

- 원본 문서 100건 파싱 완료
- HWP/PDF 파서 비교 진행
- 파싱 결과 JSON 저장 완료
- 청킹 전략 1차 비교 수행
- 품질 점검용 샘플 확인 진행

현재 기준으로 확인한 내용은 아래와 같다.

- HWP는 `olefile` 기반 파서가 가장 안정적이었음
- PDF는 `PyMuPDF`를 기본값으로 사용하는 것이 적절했음
- 청킹은 `langchain_recursive + chunk_size=1000 + overlap=200` 조합이 1차 기준으로 가장 무난했음
- CSV 축약 텍스트보다 원본 파싱 결과가 더 풍부해서 이후 검색/청킹은 parsed JSON 기준으로 진행하는 것이 적절했음

즉, 이 부분은 Track A 전체를 대신 맡아서 진행했다기보다는, Track B 화면에서 문서 결과를 안정적으로 연결하고 다음 단계로 넘기기 위해 선행 확인한 작업으로 보면 된다.
