# 자동채점 엔진 + LLM Judge 축소셋 설명

> 작성 목적: 윤민님께 현재 평가 체계가 무엇인지, 왜 2개로 나눠서 운영하는지, 실제로 어떻게 실행/해석하는지 쉽게 설명드리기 위한 문서
> 작성 기준: `scripts/auto_grader.py`, `scripts/llm_judge.py`, `자동평가_로직_구현계획.md`, `LLM_Judge_축소셋_설계안.md`

---

## 1. 한 줄 요약

현재 평가는 2개로 나뉘어 있습니다.

1. `auto_grader.py`
단답형 + 객관식처럼 정답이 비교적 명확한 문항을 자동으로 맞다/틀리다 채점하는 엔진입니다.

2. `llm_judge.py`
서술형, 비교형, 멀티턴, 거부응답처럼 Exact Match로 평가하기 어려운 답변 품질을 GPT-5 Judge가 4개 지표로 평가하는 엔진입니다.

즉,

- `auto_grader` = 정량 평가
- `LLM Judge` = 정성 평가

로 이해하시면 됩니다.

---

## 2. 왜 2개로 나눴는가

### 2-1. 자동채점 엔진만으로 충분한 경우

예를 들어 이런 문항은 정답이 비교적 명확합니다.

- 총 예산은 얼마인가
- 공고일은 언제인가
- 계약 방식은 무엇인가
- 보기 4개 중 정답은 무엇인가

이런 문항은 모델이 답한 결과를 정답과 비교해서 `정답/오답` 판정이 가능합니다.

### 2-2. 자동채점 엔진만으로 어려운 경우

반면 아래 같은 질문은 Exact Match만으로 평가하기 어렵습니다.

- 해당 사업의 목적과 구축 범위를 정리해줘
- A 사업과 B 사업을 비교해줘
- 그중 콘텐츠 관리 요구사항만 더 자세히 설명해줘
- 문서에 없는 실제 완료일을 알려줘

이런 질문은 정답 표현이 여러 가지가 가능하고, 답변이 문서 근거에 충실한지까지 봐야 해서 단순 문자열 비교로는 부족합니다.

그래서 평가를 아래처럼 분리했습니다.

- `500문항/340문항 문제은행` 같은 대량 문항: `auto_grader.py`
- 대표 서술형/비교형/거부형 샘플: `llm_judge.py`

## 3. 자동채점 엔진이 하는 일

파일: `scripts/auto_grader.py`

### 3-1. 입력

윤민님이 만들어주신 문제은행 JSON을 입력으로 받습니다.

문항 예시는 대략 아래 구조입니다.

```json
{
  "id": "DOC001_Q001",
  "document_id": "DOC001",
  "document_name": "...hwp",
  "question_type": "short_answer",
  "difficulty": "basic",
  "category": "common",
  "question": "해당 사업의 총 예산은 얼마인가?",
  "answer": "352,000,000원"
}
```

### 3-2. 실행 흐름

```text
문제은행 JSON 로드
→ 각 문항의 question 추출
→ 현재 RAG 파이프라인에 질문 투입
→ 모델 답변 생성
→ 정답(answer)와 비교
→ 문항별 결과 저장
→ 전체/유형별/난이도별 집계
```

### 3-3. 채점 규칙

현재 자동채점 엔진은 단순히 “문자열 완전 일치만” 보는 게 아닙니다. 아래 예외를 반영했습니다.

1. 공백 무시
- 예: `제한 경쟁 입찰`과 `제한경쟁입찰`은 같게 봅니다.

2. 객관식 번호 폴백
- 예: 모델이 `2번`, `3`, `두번째`, `3. 협상에 의한 계약`처럼 답해도 보기 텍스트로 매핑합니다.

3. 금액 정규화
- 예: `352,000,000원`과 `3억 5,200만원`은 같은 값으로 봅니다.
- 대신 `약 3억`, `대략 3억 5천`처럼 부정확한 표현은 인정하지 않습니다.

4. 날짜 정규화
- 예: `2024-05-26`, `2024.05.26`, `2024년 5월 26일`은 같은 날짜로 봅니다.
- 다만 `20240526`처럼 구분자 없는 형태는 현재 기준으로는 안전하지 않습니다.

5. 부분 매칭 제한
- 짧은 단어 때문에 과하게 정답 처리되는 것을 막기 위해, 현재는 4글자 이상일 때만 제한적으로 부분 매칭을 허용합니다.

6. 숫자 목록 비교
- 예: 정답이 `20, 70`일 때 답변에 두 숫자가 모두 있어야 맞게 처리합니다.

### 3-4. 데이터 품질 검사

문제은행을 로드할 때 구조 이상도 경고로 알려줍니다.

- 객관식인데 `choices`가 비어 있는 경우
- `answer`와 보기 텍스트가 정확히 안 맞는 경우
- `id`와 `document_id`가 불일치하는 경우

즉, 채점 전에 문제은행 자체 품질도 같이 확인하는 구조입니다.

### 3-5. 오류가 나면 어떻게 되는가

문항 하나에서 검색/API 오류가 나더라도 전체 배치가 멈추지 않도록 처리했습니다.

- 해당 문항은 `[오류: ...]` 형태로 기록
- 그 문항은 오답 처리
- 나머지 문항은 계속 실행

즉, 340문항이나 500문항처럼 큰 배치를 돌릴 때 중간 중단 위험을 줄였습니다.

### 3-6. 출력

자동채점 엔진 실행 후 아래 파일이 저장됩니다.

- `eval_detail.csv`: 문항별 결과
- `eval_summary.csv`: 전체/유형별/난이도별 집계

### 3-7. 현재 확인된 결과

검증 모드 기준 (75문항):

- 전체 75문항: `75/75 = 100%`
- 객관식 46문항: `46/46 = 100%`
- 단답형 29문항: `29/29 = 100%`

실제 RAG 실행 기준으로는 이전 점검에서 `69/75 = 92%` 결과를 확인했고, 이후 버그 수정/예외 처리/채점 기준 보완을 마쳤습니다.

### 3-8. 최종 340문항 평가 결과 (2026-04-17 확정)

전체 정확도: **290 / 340 = 85.29%** (오답 50문항)

유형별 결과:

- 객관식(multiple_choice): 209 / 233 = `89.70%`
- 단답형(short_answer): 81 / 107 = `75.70%`

난이도별 결과:

- basic: 100 / 112 = `89.29%`
- intermediate: 104 / 130 = `80.00%`
- advanced: 86 / 98 = `87.76%`

카테고리별 결과:

- common: 153 / 182 = `84.07%`
- specific: 131 / 150 = `87.33%`

주의할 점은, 자동채점 결과는 `현재 파이프라인 기준` 성능이라는 점입니다. 검색 방식이나 프롬프트가 바뀌면 수치도 달라질 수 있습니다.

## 4. LLM Judge 축소셋이 하는 일

파일: `scripts/llm_judge.py`

### 4-1. 왜 별도 축소셋이 필요한가

윤민님 문제은행은 주로 단답형/객관식이라 자동채점과 잘 맞습니다.

하지만 실제 RAG 품질을 보려면 아래 능력도 확인해야 합니다.

1. 문서 내용을 근거로 잘 설명하는가
2. 여러 문서를 비교/종합할 수 있는가
3. 이전 질문 맥락을 기억하고 후속 질문에 답하는가
4. 문서에 없는 정보는 제대로 거부하는가

이 부분은 Exact Match로 평가하기 어려워서, 대표 문항 20개를 따로 만든 것이 `LLM Judge 축소셋`입니다.

### 4-2. 축소셋 구성

현재 축소셋은 총 20문항입니다.

- `single_doc` 5문항
- `comparison` 5문항
- `follow_up` 6문항
- `rejection` 4문항

즉, 전체 문서/문항을 모두 평가한 것이 아니라, 대표 패턴을 확인하기 위한 소규모 진단셋입니다.

중요한 점은 이 20문항이 `75문항에서 랜덤 추출한 세트`는 아니라는 점입니다.

- 기존 문제은행
- 45문항 검수 결과
- 실제 문서 특성

을 참고해서 우리가 `의도적으로` 구성한 대표 평가셋입니다.

### 4-3. 실행 흐름

```text
축소셋 20문항 로드
→ 각 문항을 실제 RAG 파이프라인으로 실행
→ 검색 문맥 + generation 모델 답변 수집
→ GPT-5 Judge가 4개 지표로 평가
→ 문항별 점수와 판정 저장
→ 유형별 집계 및 리포트 생성
```

### 4-4. 현재 모델 운영 기준

이 부분은 헷갈릴 수 있어서 두 가지를 나눠서 설명드리는 게 정확합니다.

1. **팀에서 운영적으로 보는 모델 구도**
- `LLM Judge`는 `gpt-5`로 평가
- generation 모델은 보통 `gpt-5-mini`를 baseline처럼 볼 수 있음
- generation을 `gpt-5`로 올렸을 때 성능이 가장 좋게 나오는 방향을 기대

2. **이번 문서에서 실제로 비교 실험한 모델**
- Judge: `gpt-5` 고정
- baseline generation: `gpt-4o-mini`
- improved generation: `gpt-5`

즉, 이번 보고서에서 `baseline`이라고 적는 값은  
**실제로 실행 결과를 확보한 `gpt-4o-mini` 기준 비교**라고 이해하시면 됩니다.

현재 로컬 코드 기준 최종 설정은 아래와 같습니다.

- Judge 모델: `gpt-5`
- generation 모델: `gpt-5`

즉, 실제 최종 실행은 `gpt-5 generation + gpt-5 Judge` 조합으로 진행했습니다.

### 4-5. Judge가 보는 4개 지표

1. `Faithfulness`
- 문서에 없는 내용을 꾸며내지 않았는가

2. `Completeness`
- 질문이 요구한 핵심 요소를 빠뜨리지 않았는가

3. `Groundedness`
- 답변이 문서 근거와 직접 연결되는가

4. `Relevancy`
- 질문에 직접 답하고 있는가

각 지표는 1~5점으로 채점합니다.

### 4-6. 문항 유형별로 무엇을 보나

#### A. single_doc
- 한 문서에서 목적, 범위, 요구사항, 예산 등을 제대로 정리하는지 확인

#### B. comparison
- 두 개 이상 문서를 비교할 때 기준이 흔들리지 않는지 확인

#### C. follow_up
- 이전 턴 질문과 답변을 기억한 상태로 후속 질문을 이어서 답하는지 확인

#### D. rejection
- 문서에 없는 실제 결과나 외부 정보를 물었을 때, 억지로 추정하지 않고 “문서에서 찾을 수 없다”고 말하는지 확인

### 4-7. 거부형(rejection)이 중요한 이유

거부형은 특히 중요합니다.

예를 들면:

- 문서에는 `실제 완료일`이 없는데 계획 기간만 있음
- 문서에는 `공동수급 규칙`만 있는데 실제 참여 컨소시엄 정보는 없음
- 문서에는 기존 URL만 있는데 신규 서비스 URL은 없음

이럴 때 잘못된 RAG는 문서 속 비슷한 정보를 끌어와 “그럴듯하게” 답해버립니다.

그래서 거부형 문항은 환각 방지 품질을 보는 핵심 테스트입니다.

### 4-8. LLM Judge도 오류가 나면 어떻게 되는가

Judge 엔진 역시 중간 오류가 나더라도 전체 배치가 중단되지 않도록 만들었습니다.

- RAG 호출 실패 시: 해당 문항은 오류 메모와 함께 낮은 점수 처리
- Judge 호출 실패 시: 해당 문항은 최저점 처리 후 다음 문항으로 진행

즉, 20문항 전체를 안정적으로 끝까지 돌리는 데 초점을 맞췄습니다.

### 4-9. 출력

LLM Judge 실행 후 아래 파일이 생성됩니다.

- `judge_detail.csv`: 문항별 점수/답변/메모
- `judge_summary.csv`: 유형별 집계
- `judge_report.md`: 사람이 읽기 쉬운 요약 보고서

### 4-10. 현재 확인된 결과

LLM Judge는 **Judge를 `gpt-5`로 고정**하고,  
generation만 바꿔 비교하는 방식으로 보는 것이 가장 해석이 쉽습니다.

이번에 실제로 확인한 비교 결과는 아래와 같습니다.

| generation 모델 | PASS | REVIEW | FAIL | 평균 점수 |
|---|---:|---:|---:|---:|
| baseline `gpt-4o-mini` | 6 | 2 | 12 | 3.34 |
| improved `gpt-5` | 6 | 4 | 10 | 3.80 |

즉,
- Judge는 동일하게 `gpt-5`
- 질문도 동일한 20문항 축소셋
- 바뀐 것은 generation 모델뿐

이 조건에서 `gpt-5 generation`이 평균 점수 기준 더 좋은 결과를 보였습니다.

유형별 평균 점수는 아래와 같습니다.

| 유형 | baseline `gpt-4o-mini` | improved `gpt-5` |
|---|---:|---:|
| `single_doc` | 2.30 | 3.60 |
| `comparison` | 2.75 | 3.20 |
| `follow_up` | 3.75 | 4.00 |
| `rejection` | 4.75 | 4.50 |

해석하면:
- `single_doc`, `comparison`, `follow_up`에서는 `gpt-5`가 개선
- `rejection`은 두 모델 모두 강한 편이고, baseline이 약간 더 높게 나옴
- 전체적으로는 `gpt-5`가 더 안정적인 품질을 보였다고 해석 가능

참고로 이전에 `gpt-4o-mini Judge`로 봤을 때는 `20/20 PASS`, avg `4.99`가 나왔는데,  
이 값은 Judge 자체가 더 관대한 기준이어서 나온 결과에 가깝습니다.

그래서 지금은 `gpt-5 Judge` 기준 결과를 더 신뢰하는 것이 적절합니다.

특히 `judge_detail.csv`를 보면 감점 이유가 단순 오답이라기보다 아래 유형에 집중되어 있습니다.

- 문서에 없는 세부 항목을 답변에 추가한 경우
- 문서에 있는 핵심 항목을 일부 누락한 경우
- 검색 문맥에 없는 내용을 일반 상식처럼 보강한 경우
- 비교형 문항에서 비교 축은 맞지만, 문서 근거가 약한 내용을 함께 쓴 경우

반면 `rejection` 유형은 두 모델 모두 비교적 잘 유지되었습니다.

확인 결과:

- 문서에 없는 실제 컨소시엄 구성 정보 → 찾을 수 없다고 응답
- 실제 완료일 질문 → 계획 기간과 실제 완료일을 구분해서 거부 응답
- 검수/납품 확인 정보 → 찾을 수 없다고 응답
- 실제 투입 인원 수 → 찾을 수 없다고 응답

즉, `gpt-5 Judge` 기준에서도 거부형은 비교적 안정적으로 유지되었고, 실제로 취약한 영역은 `comparison`과 일부 `single_doc`/`follow_up`의 근거성·완전성으로 보입니다.

## 5. 자동채점 엔진과 LLM Judge의 관계

둘은 경쟁 관계가 아니라 역할이 다릅니다.

### 자동채점 엔진이 잘 보는 것

- 예산
- 날짜
- 계약 방식
- 제출 요건
- 객관식 보기 정답

즉, `정답이 비교적 명확한 문제`

### LLM Judge가 잘 보는 것

- 서술형 요약
- 비교형 질문
- 후속 질문
- 환각 방지/거부응답

즉, `답변 품질 자체를 봐야 하는 문제`

정리하면:

```text
auto_grader = "정확히 맞았는가"
LLM Judge  = "답변 품질이 좋은가"
```

## 6. 결과를 해석할 때 주의할 점

### 6-1. 현재 결과는 “현재 파이프라인 기준” 성능

지금 나온 점수는 현재 사용 중인 파이프라인 기준 결과입니다.

즉 아래 요소가 바뀌면 결과도 바뀔 수 있습니다.

- 검색 방식
- chunk 크기
- top-k 개수
- 프롬프트
- 모델
- 문서 필터링 방식
- 멀티턴 히스토리 처리 방식

따라서 지금 결과는 “절대 성능”이라기보다, `현재 파이프라인이 이 평가셋에서 얼마나 잘 동작하는가`를 보여주는 지표입니다.

### 6-2. 새 문제를 준다고 자동 학습되지는 않음

새 문항이나 새 문제은행을 추가한다고 모델이 자동으로 학습되는 구조는 아닙니다.

새 데이터의 의미는 아래와 같습니다.

- 현재 파이프라인을 새 데이터로 다시 검증
- 새로운 오답 패턴 발견
- 필요하면 채점 규칙/프롬프트/검색 방식을 개선

즉, `자동 학습`이라기보다 `추가 평가와 보완`에 가깝습니다.

## 7. 윤민님 입장에서 가장 중요하게 보시면 되는 포인트

1. 문제은행 JSON 품질
- `question_type`, `choices`, `answer`, `document_id`가 맞게 들어가야 자동채점이 안정적으로 동작합니다.

2. 정답 형식 통일
- 날짜는 `YYYY-MM-DD`, `YYYY.MM.DD`, `YYYY년 M월 D일`처럼 구분자 있는 형식이 안전합니다.
- 금액도 되도록 일관된 표기를 쓰는 것이 좋습니다.

3. 객관식 보기 품질
- 보기 2개 정도가 유사한 것은 가능하지만, 문서 근거로 명확히 구분 가능해야 합니다.
- 표현만 애매하게 비슷하면 실제 품질보다 함정문제가 될 수 있습니다.

4. 자동채점과 LLM Judge는 용도가 다름
- 자동채점이 잘 되는 문항은 Exact Match 쪽으로
- 서술형/비교형/거부형은 LLM Judge 쪽으로 보는 것이 적절합니다.

5. 결과는 파이프라인 비교용으로 재사용 가능
- 이후 팀 통합 파이프라인이 완성되면 같은 20문항 Judge 셋으로 다시 돌려서 성능 비교가 가능합니다.

## 9. 340문항/추가 문제은행 수령 시 권장 흐름

> **최종 결과 (2026-04-17 확정)**: 340문항 전체 정확도 **85.29%** (290/340)

340문항 기준 아래 순서가 가장 안전합니다.

1. JSON 구조 확인
- `items` 존재 여부
- `question_type` / `answer` / `choices` / `document_id` 확인

2. 자동채점 엔진 실행
- `scripts/auto_grader.py --모드 rag`

3. 결과 확인
- 전체 정확도
- 유형별 정확도
- 오답 문항 리스트 확인

4. 오답 패턴 분류
- 검색 실패
- 답변 생성 실패
- 채점 규칙 부족
- 문제은행 정답/보기 구조 이슈

5. 필요 시 엔진 보완
- 새로운 숫자/날짜 패턴
- 객관식 보기 응답 패턴
- 문제은행 구조 수정

즉, 340문항이 오면 우선 `auto_grader`로 전체를 통으로 돌리고, 이후 오답 패턴을 보고 보완하는 방식이 적절합니다.

## 10. 최종 정리

현재 체계는 아래처럼 이해하시면 가장 정확합니다.

- `auto_grader.py`
대량 문항을 빠르게 정량 평가하기 위한 자동채점 엔진

- `llm_judge.py`
서술형/비교형/멀티턴/거부응답처럼 Exact Match로 보기 어려운 품질을 정성 평가하기 위한 보조 평가 엔진

둘을 함께 써야,

- 대량 문제은행에 대한 자동 정확도
- 실제 RAG 품질에 대한 정성 평가

를 동시에 볼 수 있습니다.

즉, 하나는 “채점기”, 다른 하나는 “품질 진단기”라고 생각하시면 됩니다.
