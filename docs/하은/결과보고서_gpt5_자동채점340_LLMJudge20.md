## 2.2. 채점 설계 구조

### (B) Evidence 기반 문제 설계

모든 문제는 다음과 같은 근거 구조를 포함하도록 설계하였다.
- `**page**`: 근거가 위치한 페이지
- `**section**`: 해당 내용의 소제목 혹은 섹션
- `**quote**`: 문서 내 실제 근거 문장

#### 설계 의도
- 정답의 **근거 가능성 분류**
- `**grounding**`: 근거기반 평가가 가능
- Hallucination(환각) 여부를 판단할 수 있는 기준 제공

### (C) 모델 전처리 문제 구조

실제 실행에서 다음과 같은 핵심 인사이트를 확인하였다.
- 선택지가 없거나 번호만 있으면 오답 처리 불가능 → 예외 처리 필요
- 선택지가 있으면 번호만으로도 정답 판별 가능 → 유형별 로직 분리 필요
- 번호(1~5)로 대답하는 문제에서도 선택지 파싱 후 텍스트 매핑이 필수

### (D) 문제 기준 안정성

Auto Grader는 아래 목적을 달성하기 위하여 다음 기준으로 설계하였다.
- 정답이 명확한 문항만 포함, 모호성(ambiguity) 제거 → 평가 기준 일관성 확보
- LLM 기반 평가: 근거 문장에서 **만족적으로 실행 가능한 구조**를 확보하였다.

## 2.5. 자동채점 오답 50문항 패턴 분석

오답 50문항을 실제 답변 기준으로 분류해보면, 크게 5가지 패턴이 보였다.

### 패턴 A. 형식은 거의 맞지만 Exact Match에서 탈락한 경우

대략 `17건`

세부:
- 객관식 형식 차이 `15건`
- 단답형 형식 차이 `2건`

대표 예시:
- `DOC009_Q005`
  - 정답: `55개`
  - 모델답: `3. 55개`
- `DOC020_Q004`
  - 정답: `9부`
  - 모델답: `4. 9부`
- `DOC019_Q004`
  - 정답: `14일`
  - 모델답: `14일 이내`
- `DOC020_Q005`
  - 정답: `30분`
  - 모델답: `최대 30분 (발표 20분, 질의응답 10분)`

해석:
- 의미상 맞았지만, 현재 채점 기준에서는 불필요한 부연 설명이나 번호/단위가 섞여 오답 처리된 경우가 많았다.
- 특히 단답형에서 `정답 + 설명` 형태로 답하면 맞는 정보여도 탈락하는 사례가 있었다.

### 패턴 B. 숫자/날짜/표현 차이로 틀린 경우

대략 `14건`

대표 예시:
- `DOC002_Q002`
  - 정답: `2개`
  - 모델답: `2개 언어(한국어, 영어)`
- `DOC008_Q001`
  - 정답: `49,500,000원`
  - 모델답: `49,500천원`
- `DOC013_Q005`
  - 정답: `50`
  - 모델답: 정보 없음 응답

해석:
- 숫자 정규화는 들어가 있지만, `2개` vs `2개 언어`, `원` vs `천원`, 기간/범위 표현 등은 여전히 민감한 편이다.
- 일부는 채점 규칙 추가로 완화할 수 있고, 일부는 generation이 더 단답형에 맞게 답하도록 프롬프트를 조정할 수 있다.

### 패턴 C. 문서 근거는 찾았지만 요약 방향이 빗나간 경우

대략 `9건`

대표 예시:
- `DOC010_Q004`
  - 질문: `제안서는 어떤 형식으로 작성해야 하는가?`
  - 정답: `한글, 파워포인트`
  - 모델답: 형식 외에 익명화, 목차, 서류 요건까지 길게 설명
- `DOC015_Q003`
  - 질문: `제안서 평가방식으로 올바른 것은 무엇인가?`
  - 정답: `서면평가`
  - 모델답: 평가위원회 구성, 90:10 비율 등 상세 설명

해석:
- RAG가 문서를 잘 읽었는데도, 질문이 요구한 “짧고 정확한 정답” 대신 설명형 답변을 생성하면서 오답이 발생했다.
- 이건 retrieval 문제보다 generation 스타일 문제에 가깝다.

### 패턴 D. 객관식을 실제로 잘못 고른 경우

대략 `9건`

대표 예시:
- `DOC006_Q003`
  - 정답: `10월`
  - 모델답: `11월`
- `DOC009_Q003`
  - 정답: `기술 80%, 가격 20%`
  - 모델답: `기술 90%, 가격 10%`
- `DOC017_Q004`
  - 정답: `64개`
  - 모델답: `60개`
- `DOC028_Q002`
  - 정답: `공동수급 및 하도급 모두 허용`
  - 모델답: `공동수급 및 하도급 모두 금지`

해석:
- 이 유형은 단순 형식 문제가 아니라 실제로 근거를 잘못 찾았거나, 유사 보기를 헷갈린 경우다.
- 객관식 성능이 전반적으로 높긴 하지만, 비교적 비슷한 보기에서는 여전히 실수가 난다.

### 패턴 E. 과도한 정보 없음 응답

`1건`

대표 예시:
- `DOC089_Q001`
  - 정답: 사업명
  - 모델답: `문서089.hwp가 컨텍스트에 없어 확인 불가`

해석:
- 이건 RAG 검색/문서 연결이 제대로 안 된 사례로 보인다.
- 단일 문서 필터링 또는 문서명 매핑을 다시 확인할 필요가 있다.

## 2.6. 자동채점 결과 해석

정리하면, 자동채점 전체 340문항 성능은 아래처럼 볼 수 있다.

- 전체 정확도 `85.29%`, `basic`·`advanced` 구간에서 `87%~89%` 수준으로 안정적
- 다만 `intermediate`에서 오답이 집중
- 단답형 `75.70%`로 상대적으로 낮음

### 잘 된 점

- 객관식 정확도 `89.70%`로 안정적
- `basic`, `advanced`는 `87%~89%` 수준으로 양호
- 전체 340문항을 실제 API 기반으로 끝까지 실행 완료

### 한계점

- 단답형 `75.70%`로 상대적으로 낮음 (단 장 나오는 이유)
- **형식 탈락**은 실질 정답 비중이 높지만 (ex. 사업일정 관련 / 요구사항 분류 문제)
- **`intermediate` 구간**은 취약 구간으로, 단답/객관 모두 오답 비중 높음
- 즉, retrieval 문제보다 generation 스타일 문제가 주요 원인

### 개선 방향

1. 단답형 generation을 더 짧고 정답형으로 제한하도록 개선
2. `정답 + 부연설명` 형태를 더 많이 흡수하는 채점 규칙 보완
3. 숫자/기간/단위 표현 정규화 보강
4. 일부 객관식 오답은 retrieval 또는 보기 간 구분 로직 재확인

주요 이유:

주요 이유:
- 멀티턴 질문에서 이전 턴 맥락은 따라가지만, 세부 설명 시 문맥 밖 내용을 추가
- 요구사항 코드나 세부 프로세스를 과도하게 보강

대표 예시:
- `LJ_C01`, `LJ_C02`

주요 이유:
- 거부 자체는 했지만, 문서에 없는 구체 수치/절차를 참고정보처럼 덧붙임

대표 예시:

## 8. LLM Judge 결과 해석

이번 결과는 아래처럼 해석하는 것이 적절하다.

### 잘 된 점

- Judge를 `gpt-5`로 고정한 비교 실험에서 `gpt-5 generation`이 baseline보다 더 좋은 점수를 기록
- `rejection` 평균 `4.50`으로 환각 방지 성능은 비교적 안정적
- `follow_up` 평균 `4.00`으로 멀티턴 대응도 일정 수준 유지
- 질문 적합성(`Relevancy`)은 전반적으로 높음

### 취약한 점

- `comparison`이 가장 약함
- `single_doc`도 문서 요약은 하지만, 세부에서 과추정이 섞임
- 즉 “답변 방향은 맞는데, 문서 근거보다 더 많이 말하는 경향”이 가장 큰 문제
- `rejection`은 강했지만, 이 유형 하나만으로 전체 서술형 품질을 대표하긴 어려움

### 우선 개선 후보

1. generation 프롬프트에서 “문서에 없는 세부는 절대 보강하지 말 것” 강화
2. comparison 유형에서 문서 수집 범위/비교 축 명시 강화
3. follow_up에서 이전 턴 맥락 유지 + 새로운 근거 추출 균형 조정
4. rejection 응답 시 “정보 없음 + 최소한의 근거”까지만 말하도록 제한
