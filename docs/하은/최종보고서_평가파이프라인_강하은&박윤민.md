# 최종보고서(강하은, 박윤민)
# 1. 평가 파이프라인 구축

## 1.1 평가 목적 및 설계 방향

본 프로젝트의 평가는 단순 정답 일치 여부를 넘어,  
RAG 시스템의 실제 활용 가능성과 응답 품질을 종합적으로 검증하는 것을 목표로 한다.

이를 위해 평가 체계를 다음과 같이 이원화하였다.

- **정량 평가 (Auto Grader)**: 정답 기반 자동 채점
- **정성 평가 (LLM Judge)**: 의미 기반 품질 평가

즉, 정확성(Accuracy)과 응답 품질(Quality)을 분리하여 평가하는 구조로 설계하였다.

---

## 1.2 전체 평가 구조

┌─────────────┬─────────┬──────────────────────┐  
│    구분     │ 문항 수 │      평가 방식       │  
├─────────────┼─────────┼──────────────────────┤  
│ auto_grader │ 340문항 │ Exact Match 자동채점 │  
├─────────────┼─────────┼──────────────────────┤  
│ LLM Judge   │ 20문항  │ GPT 기반 품질 비교   │  
└─────────────┴─────────┴──────────────────────┘  

- **Auto Grader**: 전체 성능을 측정하는 메인 평가
- **LLM Judge**: 정답 기반 평가가 어려운 유형에 대한 보완 평가이자, generation 모델 비교용 품질 평가

---

# 2. Auto Grader 설계

## 2.1 개요

Auto Grader는 총 340문항으로 구성된 문제은행을 기반으로,  
모델의 응답을 **Exact Match 방식**으로 자동 채점한다.

## 2.2 평가 대상

- 객관식 (Multiple Choice): 233문항
- 단답형 (Short Answer): 107문항

→ 명확한 정답이 존재하는 문제 유형

## 2.3 평가 방식

모델 응답과 정답을 문자열 단위로 비교하며, 아래 6단계를 순서대로 적용한다.

```
1. 객관식 번호 매핑     "2번" / "2. 제한경쟁입찰" → 보기 텍스트로 변환
2. 공백 제거 Exact Match  "일반 경쟁 입찰" == "일반경쟁입찰" → 정답
3. 부분 포함 체크       정답이 4글자 이상일 때, 답변 내 포함 여부 확인
4. 숫자 정규화 비교     "49,500천원" == "49,500,000원" → 정답
5. 날짜 정규화 비교     "2024.07.10" == "2024년 7월 10일" → 정답
6. 숫자 목록 비교       정답 "20, 70" → 20과 70 모두 포함 시 정답
```

## 2.4 역할

- 전체 시스템의 **기본 성능(Accuracy)** 측정
- 대규모 문항 기반 **안정적인 정량 평가 수행**
- 모델의 정보 검색 및 추출 능력 검증
- 평가 결과를 통해 모델 개선 방향의 기준점 제공

---

# 3. LLM Judge 설계

## 3.1 도입 배경

Exact Match 기반 평가의 한계:

- 표현 방식이 다른 경우에도 오답 처리됨
- 서술형, 비교형 등 복합 질의 평가 불가능
- 부분 정답이나 의미적 일치 여부 판단 불가

이러한 한계를 보완하기 위해  
**LLM 기반 품질 평가(LLM Judge)**를 도입하였다.

---

## 3.2 평가 데이터 구성 (축소셋)

- 총 20문항으로 구성
- 팀장님이 작성한 설계안 문서를 기반으로 제작
- 실제 DB에 저장된 RFP 문서를 참조하여 질문 생성

### 축소셋을 사용하는 이유

- 본 평가는 전체 성능 측정보다는 **품질 진단 목적**
- 설계안 자체가 "축소셋" 기반 평가를 전제로 구성됨
- Auto Grader(340문항)가 이미 정량 평가를 담당

→ 따라서 LLM Judge는 **보완적 역할** 수행

---

## 3.3 문제 유형 구성

LLM Judge는 다음 4가지 유형을 균형 있게 포함한다.

| 유형 | 문항 수 | 설명 |
|------|---------|------|
| 서술형 (single_doc) | 5 | 단일 문서 기반 서술형 질문 |
| 비교형 (comparison) | 5 | 다문서 비교 및 필터링 |
| 멀티턴 (follow_up) | 6 | 이전 답변 기억 후 드릴다운 |
| 거부응답 (rejection) | 4 | 문서에 없는 정보 → 거부해야 PASS |

→ 실제 사용자 질의에서 발생 가능한 복잡한 상황을 반영

---

## 3.4 평가 방식

LLM Judge는 **Judge 모델을 `GPT-5`로 고정**하고,  
generation 모델만 바꿔가며 응답 품질을 비교하는 방식으로 설계하였다.

즉, Judge까지 함께 바뀌면 점수 차이가 Judge 성향 때문인지 generation 품질 때문인지 구분하기 어려우므로,  
**같은 Judge / 같은 20문항 축소셋 / 같은 판정 기준**을 유지한 상태에서 generator만 비교하였다.

### 평가 지표 (각 1~5점)

- **Faithfulness**: 검색 문맥에 없는 사실을 꾸며내지 않았는가
- **Completeness**: 질문이 요구한 핵심 요소를 빠뜨리지 않았는가
- **Groundedness**: 답변이 검색 문맥에 직접 근거하고 있는가
- **Relevancy**: 질문에 직접 답하고 있는가

### 판정 기준

```
평균 4.0 이상 + Faithfulness 4 이상 → PASS
평균 3.0 미만 or Faithfulness ≤ 2   → FAIL
그 외                                → REVIEW
```

### 비교 대상

- **baseline generation**: `gpt-4o-mini`
- **improved generation**: `gpt-5`
- **Judge**: `gpt-5` 고정

→ 발표에서는 위 비교 실험을 통해 **서술형 품질 차이**를 보여주고,  
340문항 Auto Grader는 **최종 선택 모델(`gpt-5`)의 전체 성능**을 제시하는 구조로 정리하였다.

---

## 3.5 역할

- Exact Match로 평가 불가능한 문제 유형 보완
- 의미 기반 응답 품질 평가
- 사용자 관점에서의 답변 유용성 검증
- RAG 시스템의 **실제 서비스 적용 가능성 평가**

---

# 4. 평가 결과 및 피드백

## 4.1 정량 평가 결과 (Auto Grader)

> 모델: GPT-5 | 검색 모드: 하이브리드 | 문항 수: 340

| 구분 | 정답 | 전체 | 정확도 |
|------|------|------|--------|
| 전체 | 290 | 340 | **85.29%** |
| 객관식 | 209 | 233 | 89.70% |
| 단답형 | 81 | 107 | 75.70% |

**난이도별 경향**
- intermediate 구간에서 오답이 가장 많이 발생

---

## 4.2 정성 평가 결과 (LLM Judge)

> 비교 조건: Judge = GPT-5 고정 | 문항 수: 20

| generation 모델 | PASS | REVIEW | FAIL | 평균 점수 |
|------|------:|------:|------:|------:|
| baseline (`gpt-4o-mini`) | 6 | 2 | 12 | 3.34 |
| improved (`gpt-5`) | 6 | 4 | 10 | **3.80** |

→ Judge를 고정한 상태에서 generation만 `gpt-5`로 바꾸었을 때 평균 점수가 **3.34 → 3.80**으로 개선되었다.

### 유형별 평균 점수 비교

| 유형 | baseline (`gpt-4o-mini`) | improved (`gpt-5`) |
|------|------:|------:|
| single_doc | 2.30 | **3.60** |
| comparison | 2.75 | **3.20** |
| follow_up | 3.75 | **4.00** |
| rejection | **4.75** | 4.50 |

**유형별 경향**
- `rejection(거부응답)`은 두 모델 모두 상대적으로 강한 편
- `comparison(비교형)`은 두 모델 모두 가장 약한 유형
- `gpt-5 generation`으로 올렸을 때 `single_doc`, `comparison`, `follow_up`에서 개선이 확인됨

---

## 4.3 오답 패턴 분석

### Auto Grader 오답 패턴

| 패턴 | 설명 |
|------|------|
| 답변 길이 문제 | 정답은 맞지만 설명형으로 길게 답해 Exact Match 탈락 |
| 표현 차이 | 숫자·날짜·단위 표현 방식 불일치 |
| 스타일 불일치 | 질문이 요구한 짧은 정답 대신 설명형으로 답변 |
| 오선택 | 일부 객관식에서 실제로 다른 보기를 선택 |

→ retrieval 자체보다 **generation 답변 스타일**로 인한 감점 비중이 높음

### LLM Judge 오답 패턴

| 지표 | 경향 |
|------|------|
| Faithfulness / Groundedness | 문서에 없는 세부 내용을 덧붙이는 경향으로 감점 |
| 비교형·서술형 | 문서 근거보다 과하게 설명하는 패턴 빈번 |

→ 즉, 질문 적합성 자체는 높지만 문서 근거를 벗어난 보강 설명 때문에 감점되는 경우가 많았다.

---

## 4.4 종합 분석 및 개선 방향

### 종합 분석

- **Auto Grader**: 340문항 전체에서 `gpt-5` 기준 **85.29%**로, 객관식·정답형 문항에서는 비교적 안정적으로 동작
- **LLM Judge**: Judge를 `gpt-5`로 고정한 비교 실험에서 `gpt-5 generation`이 baseline보다 더 좋은 품질을 보였음
- 다만 서술형·비교형에서는 여전히 문서 근거 중심의 보수적 응답이 필요

→ 즉, 본 발표에서는  
`LLM Judge = baseline vs gpt-5 비교`,  
`340문항 Auto Grader = gpt-5 최종 성능`  
구조로 결과를 해석하는 것이 가장 타당하다.

### 개선 방향

- **Generation 스타일 조정**: 짧고 보수적인 답변 유도 (프롬프트 튜닝)
- **Grounding 강화**: 문서 근거 없는 추가 설명 억제
- **Retrieval 정확도 향상**: Chunking 전략 개선
- **Hallucination 감소**: 근거 없는 세부 사항 생성 방지
- **멀티턴 대응 로직 고도화**: 이전 맥락 유지 품질 개선

---

# 5. 결론

본 평가 파이프라인은 다음과 같은 특징을 가진다.

- 대규모 자동 채점 기반의 **정량 평가 체계** (Auto Grader, 340문항, 85.29%)
- Judge를 고정하고 generation만 비교한 **정성 평가 체계** (LLM Judge, 20문항, baseline 3.34 → gpt-5 3.80)
- 두 평가 방식의 결합을 통한 **현실적인 성능 검증 구조**

현재 시스템은 객관식·정답형 문항에서 안정적으로 동작하고 있으며,  
서술형·비교형에서는 문서 근거 중심으로 더 짧고 보수적으로 답하도록  
추가 보완 여지가 있는 상태이다.

이를 통해 단순 정확도를 넘어  
실제 서비스 환경에서의 활용 가능성을 효과적으로 평가하였다.
