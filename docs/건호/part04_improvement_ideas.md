## 1. 아이디어 1 — Hybrid RRF 검색 (Dense + Sparse 결합)

### 1-1. 문제 정의

Part 04-04 분석 결과, **어떤 질문 유형에서도 "단일 방식이 최선"이 아니다**:

| TYPE | 최선 방식 | 단독 사용 시 리스크 |
|---|---|---|
| TYPE 1 | 3종 동등 | — |
| TYPE 2 | Dense | Sparse는 고려대 편향 (10개 중 6개 오매칭) |
| TYPE 3 | Sparse | Dense는 맥락 소실 드리프트 |
| TYPE 4 | Sparse | Dense는 단어 혼동 드리프트 (Q41 광주연구원 등) |

**근본 딜레마:** 운영 환경에서는 사용자 질문이 어떤 TYPE인지 **사전에 알 수 없음** → 단일 방식 선택이 불가능

### 1-2. 해결 아이디어 설명

**RRF (Reciprocal Rank Fusion, 역순위 융합):**
두 검색 방식의 결과를 **순위 기반**으로 결합하여 단일 순위 리스트 생성.

**수식:**
```
RRF_score(문서) = vector_weight × 1/(rrf_k + Dense순위)
                + bm25_weight × 1/(rrf_k + Sparse순위)

채택한 파라미터:
- vector_weight = 0.7  (Dense 가중치)
- bm25_weight   = 0.3  (Sparse 가중치)
- rrf_k         = 60   (표준값, 상위 순위 편중 완화)
```

**왜 점수가 아닌 순위를 쓰는가:**
- Dense 점수(코사인 유사도, 0~1)와 Sparse 점수(BM25, 0~수십)는 **스케일이 다름**
- 단순 가중합 시 Sparse가 수치적으로 압도 → 의도와 다른 결과
- 순위(1, 2, 3...)는 공통 스케일이라 공정한 결합 가능

### 1-3. 구현 방식

**코드 위치:** `dense_sparse_comparison.ipynb` cell-18

```python
def hybrid_search(query, top_k=5, candidate_k=10,
                  vector_weight=0.7, bm25_weight=0.3, rrf_k=60):
    # 1) 각 방식에서 상위 candidate_k개(10개) 후보 생성
    dense_results = dense_search(query, top_k=candidate_k)
    sparse_results = sparse_search(query, top_k=candidate_k)

    # 2) RRF 점수 계산
    fused = {}
    for rank, (idx, _) in enumerate(dense_results, 1):
        fused[idx] = fused.get(idx, 0.0) + vector_weight * (1.0 / (rrf_k + rank))
    for rank, (idx, _) in enumerate(sparse_results, 1):
        fused[idx] = fused.get(idx, 0.0) + bm25_weight * (1.0 / (rrf_k + rank))

    # 3) 합산 점수 기준 재정렬 후 top-k 반환
    ranked = sorted(fused.items(), key=lambda x: x[1], reverse=True)[:top_k]
    return ranked
```

**파라미터 선택 근거:**
- `vector_weight=0.7 / bm25_weight=0.3`: Dense가 의미 검색에서 더 범용적으로 유효하므로 기본 우위 부여
- `candidate_k=10`: 각 방식에서 상위 10개를 뽑아야 두 방식의 결과가 충분히 섞임 (너무 작으면 Dense top-1과 Sparse top-1만 비교되어 RRF 효과 약화)
- `rrf_k=60`: RRF 표준 논문값. 작을수록 상위 순위 편중, 클수록 완만

### 1-4. 실측 효과

### 1-6. 한계·부작용

**한계:**
1. **성능 개선이 아닌 안전한 타협** — Dense/Sparse 각자의 최고 성능보다 극적으로 뛰어나진 않음. "평균적으로 안전"이 핵심 가치
2. **가중치 고정 (0.7:0.3)** — 질문 유형별 최적 가중치가 다름에도 정적 값 사용. 예: TYPE 2는 Dense 0.9 / Sparse 0.1이 더 나을 수 있음
3. **TYPE 2의 Sparse 편향 자체는 해결하지 못함** — Hybrid는 Dense를 따라가 **회피**할 뿐, Sparse 순위에 고려대가 계속 1위로 남음 → 아이디어 2로 보완 필요

**부작용:**
1. **검색 비용 증가** — Dense 1회 + Sparse 1회 + RRF 계산 → 단일 방식 대비 약 2배 시간/비용
2. **디버깅 복잡도 상승** — 결과가 이상할 때, Dense 탓인지 Sparse 탓인지 가중치 탓인지 원인 추적이 어려움

**부작용 대응:**
- 검색 비용: 현 프로젝트 규모(5,948 청크)에서는 무시 가능한 수준. 수백만 청크로 확장 시 재평가 필요
- 디버깅: `dense_sparse_hybrid_per_question.csv`에 3종 top-1 파일명을 모두 기록 → 원인 추적 가능한 구조로 설계 완료

## 2. 아이디어 2 — BM25 문서 길이 정규화 강화

### 2-1. 문제 정의

**TYPE 2(비교·목록 질문)에서 Sparse의 치명적 편향 관찰:**

| 측정 항목 | 값 |
|---|---|
| Dense-Sparse Top-1 일치율 | **10%** (4개 TYPE 중 최저) |
| Sparse가 고려대학교를 top-1으로 반환한 질문 수 | **10개 중 6개** |
| 고려대학교 문서 청크 수 | **458개** (전체 5,948개 중 7.7%) |
| 2순위 긴 문서의 청크 수 | 약 30~50개 |

**원인:**
고려대학교 RFP 문서가 **압도적 길이 outlier**로, BM25가 어떤 추상적 질문(예: "AI 사업들", "ERP 기관들")에도 고려대를 top-1으로 반환.

**왜 이게 심각한가:**
- 사용자 관점: 질문과 전혀 관련 없는 문서가 1순위로 노출됨
- 파이프라인 관점: 후단 LLM에 잘못된 문서가 전달되어 환각·오답 유발
- 신뢰도 관점: Sparse 단독으로는 **질문 유형을 예측할 수 없는 운영 환경**에서 쓸 수 없는 상태

### 2-2. 해결 아이디어 설명

**BM25 수식 복습:**
```
BM25(D, Q) = Σ IDF(q) × (f(q,D) × (k1+1)) / (f(q,D) + k1 × (1 - b + b × |D|/avgdl))

- |D|    : 문서 길이 (토큰 수)
- avgdl  : 평균 문서 길이
- k1     : 단어 빈도 스케일링 (기본 1.5)
- b      : 길이 정규화 강도 (기본 0.75, 0~1 범위)
```

**핵심 파라미터 `b`:**
- `b=0` → 문서 길이 완전 무시 (긴 문서 유리)
- `b=1` → 문서 길이 완전 반영 (긴 문서 강한 페널티)
- `b=0.75` (기본값) → 중간 수준 페널티

**제안: `b`를 0.75 → 0.9~1.0으로 상향**
- 긴 문서(고려대 458청크)가 단순 길이만으로 얻는 유리함을 강하게 억제
- 짧은 문서도 관련 키워드가 집약되어 있으면 경쟁 가능해짐

**부가 제안: 문서(document) 단위 정규화 추가**
- 현재는 청크 단위로만 BM25 적용 중
- 문서 1개가 여러 청크를 가질 때, **문서 단위로 top-1 점수를 정규화**하는 후처리 추가
  - 예: 고려대 문서의 458개 청크 중 1등 점수만 대표값으로 쓰고, 다른 문서의 1등 점수와 공정 비교

### 2-3. 구현 방식

**방법 A: `rank_bm25` 라이브러리 파라미터 조정**

```python
from rank_bm25 import BM25Okapi

# 현재 (기본값)
bm25 = BM25Okapi(tokenized_chunks)  # b=0.75, k1=1.5

# 제안 변경
bm25 = BM25Okapi(tokenized_chunks, b=0.95, k1=1.5)
```

단 한 줄만 바꾸면 됨. 나머지 검색 로직(`sparse_search()`)은 수정 불필요.

**방법 B: 문서 단위 후처리 추가**

```python
def sparse_search_doc_normalized(query, top_k=5):
    tokens = tokenize_korean(query)
    scores = bm25.get_scores(tokens)

    # 문서별 최고 점수만 유지 (청크 중복 제거)
    doc_best = {}
    for idx, score in enumerate(scores):
        doc = chunk_files[idx]
        if doc not in doc_best or score > doc_best[doc][1]:
            doc_best[doc] = (idx, score)

    # 문서 단위 정렬
    ranked = sorted(doc_best.values(), key=lambda x: x[1], reverse=True)[:top_k]
    return ranked
```

**방법 A + B 병행 권장** — 파라미터 조정(A)은 근본적 개선, 문서 정규화(B)는 outlier 추가 방어.

### 2-4. 기대 효과

**정량 예측 (검증 전 추정):**

| 지표 | 현재 | 개선 후 (방법 A, b=0.95) | 개선 후 (A+B) |
|---|---|---|---|
| TYPE 2 D-S 일치율 | 10% | 약 40~50% | **50% 이상** |
| 고려대 top-1 반환 빈도 (TYPE 2) | 6/10 | 2~3/10 | 1/10 이하 |
| TYPE 1 Hit Rate | 1.00 | 1.00 (영향 없음) | 1.00 |
| 전체 파이프라인 신뢰도 | 낮음 | 중간 | **높음** |

**기대 동작 예시 (Q17: "ERP 시스템 관련 사업"):**

- **현재 Sparse:** 고려대학교 (오매칭, "ERP" 단어가 고려대 문서에도 조금 있음 + 458청크 이점)
- **개선 후 Sparse:** 한국가스공사 또는 인천공항운영서비스 ERP 사업 (실제 ERP 중심 문서)

### 2-5. 검증 방법

**검증 단계:**

1. **A/B 비교 실행**
   - 현재 `b=0.75` 결과 보존 (`dense_sparse_hybrid_per_question.csv`)
   - 개선 적용 후 `b=0.95` 또는 `b=1.0` 버전으로 동일 45개 질문 재실행
   - 결과 CSV 별도 저장 (예: `dense_sparse_hybrid_per_question_b95.csv`)

2. **핵심 지표 비교**
   - TYPE 2 Dense-Sparse 일치율 변화: **10% → 몇 %?**
   - 고려대 top-1 반환 빈도: 6/10 → 몇 /10?
   - TYPE 1 Hit Rate 유지 여부 (1.00 유지되어야 함)

3. **회귀 검증**
   - TYPE 3·4에서 Sparse 성능이 **저하되지 않는지** 확인
   - 기관명 명시 질문(TYPE 1, 4 일부)에서 여전히 정확한지 점검

4. **정성 검증**
   - Q15~Q20의 Sparse top-1이 고려대가 아닌 **합리적 문서**로 바뀌는지 수동 검토

**성공 기준:**
- TYPE 2 D-S 일치율 30% 이상 달성
- 고려대 편향 3건 이하 감소
- TYPE 1/3/4 성능 유지 (저하 없음)

### 2-6. 한계·부작용

**한계:**
1. **다른 outlier 문서에는 효과 제한적** — 고려대 같은 극단적 outlier만 해결. 중간 길이 문서들의 경계 케이스는 여전히 판단 어려움
2. **추상 질문 자체의 난제는 미해결** — "AI 사업들" 같은 질문은 BM25가 근본적으로 약함. Dense/Hybrid 병용 필요
3. **TYPE 3 (맥락 후속 질문)은 도움 안 됨** — 이 문제는 길이 정규화와 무관. 별도 해결(멀티턴 query rewriting) 필요

**부작용:**
1. **짧지만 덜 관련된 문서가 상위에 오를 위험** — b=1.0 극단 설정 시, 짧고 키워드만 우연히 맞는 문서가 과하게 유리해질 수 있음
2. **TYPE 1에서 영향 가능성** — 기관명 명시 질문에서 여전히 100% 성능을 유지하는지 검증 필수. 대부분의 경우 영향 없지만, 고려대 관련 TYPE 1 질문(Q02)의 경우 고려대 문서의 상대적 점수가 낮아지는 효과가 발생할 수 있음

**부작용 대응:**
- 부작용 1: `b` 값을 극단(1.0)이 아닌 **0.9~0.95** 중간값으로 선택하여 균형 유지
- 부작용 2: 검증 단계에서 TYPE 1 Hit Rate 모니터링 → 저하 시 `b` 값 재조정

## 3. 두 아이디어 통합 효과 및 우선순위

### 3-1. 통합 아키텍처

```
[사용자 질문]
    ↓
[Dense 검색] + [Sparse 검색 (b=0.95 적용)]  ← 아이디어 2
    ↓                        ↓
        [RRF 결합]           ← 아이디어 1
    ↓
[최종 top-5 문서]
    ↓
[LLM (후단 파이프라인)]
```

**상호 보완 관계:**
- 아이디어 2가 Sparse 자체의 품질을 높임 → 아이디어 1이 이 개선된 Sparse를 활용하여 **더 좋은 결합 결과** 도출
- 아이디어 1 없이 아이디어 2만 쓰면 → Sparse 단독 개선 but Dense의 장점 활용 못 함
- 아이디어 2 없이 아이디어 1만 쓰면 → 현 상태 유지 (Hybrid가 Dense를 따라가며 회피, 편향 근본 원인은 잔존)

### 3-2. 예상 통합 효과

| 지표 | Baseline (Dense 단독) | 아이디어 1만 (Hybrid) | **아이디어 1 + 2** |
|---|---|---|---|
| TYPE 1 Hit Rate | 1.00 | 1.00 | 1.00 |
| TYPE 2 D-S 일치율 | 10% | 10% (회피만) | **40% 이상** |
| TYPE 3 경향 (Sparse 활용) | 0% | 62% | 62% (유지) |
| TYPE 4 경향 (Sparse 활용) | 0% | 90% | 90% (유지) |
| Sparse 고려대 편향 | 6/10 | 6/10 (회피만) | **1~2/10** |
| 운영 환경 신뢰도 | 중 | 중~상 | **상** |

### 3-3. 우선순위 제안

| 순위 | 아이디어 | 근거 |
|---|---|---|
| **1순위** | 아이디어 1 (Hybrid RRF) | ✅ 이미 구현·검증 완료. 즉시 운영 적용 가능. 리스크 최소 |
| **2순위** | 아이디어 2 (BM25 정규화) | ⏳ 구현 필요하지만, 파라미터 한 줄 변경 수준으로 간단. 검증 후 적용 |

**적용 전략:**
1. **단기(즉시):** 아이디어 1을 현 파이프라인에 기본 검색 방식으로 채택
2. **중기(후속 작업):** 아이디어 2를 구현·검증 후, Hybrid 내부의 Sparse 부품으로 투입
3. **장기(범위 밖):** TYPE 3 맥락 문제는 **멀티턴 query rewriting**으로 별도 해결 — 이는 Retrieval 계층이 아닌 쿼리 전처리 계층 개선 과제

### 3-4. 범위 밖(향후 제안) 개선 방향

본 분석에서 드러났지만 Part 04 범위를 벗어나는 과제:

1. **멀티턴 Query Rewriting** — TYPE 3 맥락 소실 드리프트의 근본 해결책
2. **LLM 프롬프트 제약 강화** — TYPE 4 거부 질문에서 모델이 "정보 없음"을 정확히 판단하도록 하는 프롬프트 엔지니어링
3. **청크 재설계 (chunking 전략)** — 고려대 458청크 outlier의 근본 원인은 chunk 단위가 너무 잘게 쪼개진 데서 기인. 청크 크기를 키우면 전체 청크 수 감소 → 편향 완화

## 4. 결론

- **아이디어 1 (Hybrid RRF)** 은 단일 방식 선택의 딜레마를 **우회**하는 안전한 해결책으로, 이미 구현·검증 완료 상태다. 즉시 운영 적용 가능하며, 리스크 대비 효과가 가장 확실한 개선안이다.
- **아이디어 2 (BM25 길이 정규화)** 는 TYPE 2에서 관찰된 Sparse 고려대 편향의 **근본 원인**을 수정하는 접근으로, 파라미터 한 줄 변경 수준의 낮은 구현 비용 대비 TYPE 2 일치율을 10%에서 40% 이상으로 끌어올릴 것으로 예상된다.
- 두 아이디어는 **상호 보완적**이며, 함께 적용 시 Hybrid 검색이 더 우수한 Sparse 결과를 활용하게 되어 전체 파이프라인 신뢰도가 극대화된다.
- 본 분석 범위 밖이지만, **멀티턴 Query Rewriting**과 **LLM 프롬프트 제약**은 TYPE 3·4의 잔존 한계를 해결할 후속 과제로 명시해둔다.
