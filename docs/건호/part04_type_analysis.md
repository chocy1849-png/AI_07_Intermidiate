## 1. TYPE 1 — 단일 기관 사실 질문 (factual)

### 1-1. 질문 특성 + 대표 예시
- **특성:** 특정 기관명 + 구체적 사업명이 질문 안에 명시됨. "X 기관의 Y 사업에 대해 알려줘" 형태
- **대표 예시:**
  - Q01: "**국민연금공단**이 발주한 **2024년 이러닝시스템 운영 용역**의 사업 예산은 얼마야?"
  - Q05: "**한국가스공사** 차세대 **통합정보시스템(ERP)** 구축 사업의 입찰 참여 마감일이 언제야?"

### 1-2. 3종 검색 결과 동작 패턴
- **수치:** Dense Hit = Sparse Hit = Hybrid Hit = **1.00** (12/12 전부 정답)
- **Top-1 문서 일치율:** 3종 모두 완전 동일한 문서를 1순위로 가져옴
- **관찰:** 기관명이 고유명사(예: "국민연금공단")로 질문에 있으면, Sparse의 BM25 키워드 매칭과 Dense의 의미 임베딩이 같은 문서를 가리킨다

### 1-3. 원인 진단
**(a) 알고리즘 관점**
- BM25는 희귀 키워드(기관명)에 높은 IDF를 부여 → 해당 기관 문서만 top 점수
- Dense 임베딩은 기관명과 사업명이 함께 등장한 청크를 강하게 매칭 → 동일 문서 선택

**(b) 데이터 관점**
- RFP 청크는 문서 맨 앞에 `prefix_v2`로 기관명이 삽입되어 있음 → 두 방식 모두 prefix에서 매칭 성공
- 기관명이 질문과 문서에 정확히 동일한 표기(예: "국민연금공단")로 있어 표기 불일치 문제 없음

### 1-4. 권장사항 + 한계
- **권장:** 이 TYPE에서는 **어떤 방식을 써도 무방**. 계산 비용이 낮은 Sparse(BM25)만 써도 충분
- **한계:** 질문 검증용 TYPE으로만 의미 있음. 실제 운영 환경에서 난이도가 가장 낮은 질문

### 1-5. Part 04-05 개선 아이디어 연결
- 이 TYPE은 이미 100% 성능이므로 개선 여지 없음. 개선 아이디어의 **검증용 baseline**으로 활용

## 2. TYPE 2 — 비교·목록 질문 (comparison)

### 2-1. 질문 특성 + 대표 예시
- **특성:** 여러 기관 또는 조건(예산·지역·키워드)으로 다수 문서를 묶어서 찾아야 하는 질문. 단일 top-1 문서로는 답이 불충분
- **대표 예시:**
  - Q13: "**고려대학교** 차세대 포털·학사 정보시스템 사업이랑 **광주과학기술원** 학사시스템 기능개선 사업을 비교해줘" (GT = 2개 문서)
  - Q17: "**ERP 시스템** 관련 사업을 발주한 기관들을 찾아서 비교해줘" (GT 없음, 다수 기관 매칭 필요)

### 2-2. 3종 검색 결과 동작 패턴
- **수치:** Dense-Sparse 일치율 **10%** (최저), Hybrid-Dense 일치율 **100%**, Hybrid-Sparse 일치율 10%
- **Top-1 문서 패턴 (실제 결과):**

| 질문 | Dense top-1 | Sparse top-1 | Hybrid top-1 |
|---|---|---|---|
| Q15 (한국수자원공사 사업들) | (사)한국대학스포츠협의회 | **고려대학교** | (사)한국대학스포츠협의회 |
| Q16 (10억 이상 예산) | 국방과학연구소 | **고려대학교** | 국방과학연구소 |
| Q17 (ERP 기관들) | 부산관광공사 | **고려대학교** | 부산관광공사 |
| Q19 (AI/인공지능) | 서울특별시 여성가족재단 | **고려대학교** | 서울특별시 여성가족재단 |
| Q20 (2025년 마감) | 재단법인경기도일자리재단 | **고려대학교** | 재단법인경기도일자리재단 |

- **관찰:** **Sparse는 10개 질문 중 6개에서 "고려대학교"를 top-1로 반환** — 심각한 편향

### 2-3. 원인 진단
**(a) 알고리즘 관점**
- BM25의 본질: 길이 정규화(b 파라미터) 한계로, **문서가 매우 길면 다양한 키워드가 우연히 포함될 확률↑** → 어떤 일반 키워드든 매칭될 가능성 높음
- 비교·목록 질문은 **추상적 키워드("10억 이상", "ERP", "AI")**가 많음 → BM25가 의미보다 단순 빈도에 의존하여 오매칭

**(b) 데이터 관점**
- 고려대학교 문서(`고려대학교_차세대 포털·학사 정보시스템 구축사업.pdf`)는 **458개 청크**로 분할됨 (전체 5,948개 중 약 7.7%)
- 다음으로 많은 문서도 30~50청크 수준 → 고려대 문서가 **압도적 outlier**
- 긴 문서일수록 BM25 점수 합산 구조상 top-1 가능성 ↑
- prefix에 "고려대학교"가 458번 반복 등장 → 임의 키워드에 자주 히트

### 2-4. 권장사항 + 한계
- **권장: Dense 또는 Hybrid** (Hybrid가 Dense를 100% 따라감 → 안전)
- Sparse 단독 사용은 비교·목록 질문에서 **치명적**
- **한계:** Dense도 10개 중 1개 질문(Q13)만 ground_truth가 있어 완전한 Hit Rate 검증은 불가. 다만 "무엇을 top-1로 반환하는가" 관점에서 Dense가 훨씬 합리적

### 2-5. Part 04-05 개선 아이디어 연결
- **아이디어 2 (BM25 문서 길이 정규화 강화)**가 이 TYPE을 직접 해결함
  - BM25의 `b` 파라미터를 기본 0.75 → 0.9~1.0으로 증가시켜 긴 문서 페널티 강화
  - 또는 청크 단위가 아닌 **문서 단위 normalization** 추가 고려
- 개선 후 예상 효과: Sparse의 고려대 편향 완화 → Dense-Sparse 일치율 10% → 40% 이상 회복

## 3. TYPE 3 — 대화 맥락 후속 질문 (follow_up / factual)

### 3-1. 질문 특성 + 대표 예시
- **특성:** 이전 턴의 대화 맥락을 이어가는 질문. 대명사("이 사업", "그중에서", "두 사업") 또는 생략된 주어가 많음
- **대표 예시:**
  - Q24: "콘텐츠 개발 관리 요구사항에 대해서 더 자세히 알려줘" (앞 턴: 국민연금공단 이러닝)
  - Q27: "그럼 모니터링 업무에 대한 요청사항이 있는지 찾아보고 알려줘" (앞 턴: 기초과학연구원 극저온시스템)
  - Q32: "이 사업에서 외부 시스템과의 연계 요구사항이 있어?" (앞 턴: 한국원자력연구원)

### 3-2. 3종 검색 결과 동작 패턴
- **수치:** Dense-Sparse 일치율 **31%**, Hybrid-Dense 54%, Hybrid-**Sparse 62%** → Hybrid가 Sparse 쪽으로 약간 기울어짐
- **ground_truth 없음** (13개 전부) → Hit Rate 측정 불가, 정성 분석 필요

**대표 실패/성공 케이스:**

| 질문 | 앞 턴 맥락 | Dense top-1 | Sparse top-1 | Hybrid top-1 | 판정 |
|---|---|---|---|---|---|
| Q24 (콘텐츠 개발) | 국민연금공단 이러닝 | **고려대학교** (드리프트) | 국민연금공단 (맥락 유지) | 국민연금공단 | Dense 실패 |
| Q27 (모니터링) | 기초과학연구원 | **고려대학교** (드리프트) | 기초과학연구원 | 기초과학연구원 | Dense 실패 |
| Q32 (외부 연계) | 한국원자력연구원 | **고려대학교** (드리프트) | 한국발명진흥회 | 서울특별시 | 3종 모두 실패 |
| Q26 (AI 예측) | 기초과학연구원 | 기초과학연구원 | 기초과학연구원 | 기초과학연구원 | 3종 모두 성공 |

### 3-3. 원인 진단
**(a) 알고리즘 관점**
- Dense 임베딩: 질문 자체의 의미(예: "콘텐츠 개발")만으로 벡터화 → 앞 턴 맥락 정보가 임베딩에 없음 → 일반적 의미로 가장 가까운 고려대 문서로 드리프트
- Sparse: 앞 턴과 공통 키워드(예: "이러닝", "모니터링")가 있으면 같은 문서가 점수 유지 → 맥락 유지에 유리
- 현 파이프라인은 **멀티턴 재구성(query rewriting) 미적용** → 단일 질문 그대로 검색됨

**(b) 데이터 관점**
- 후속 질문의 단어가 RFP 원문 prefix에 키워드로 남아있는 경우가 많음 ("모니터링", "연계", "콘텐츠")
- Sparse가 이 키워드를 그대로 잡아 같은 문서 유지
- 반면 Dense는 해당 키워드의 **의미적 가장 가까운 문서**를 새로 찾아가면서 드리프트 발생

### 3-4. 권장사항 + 한계
- **권장: Sparse 또는 Hybrid** (맥락 유지에 키워드가 유효)
- 단, **완전한 해결은 멀티턴 query rewriting** 필요 — 앞 턴 정보를 질문에 병합한 뒤 검색
- **한계:** 현재 파이프라인 수준에서 Sparse가 Dense보다 낫지만, 이상적 해결은 retrieval 계층이 아닌 **쿼리 전처리 계층**에서 이루어져야 함

### 3-5. Part 04-05 개선 아이디어 연결
- **아이디어 1 (Hybrid RRF)**이 부분적으로 이 TYPE을 개선함 — Sparse 가중치가 커서 맥락 유지 경향을 가짐
- **근본 해결**은 이 분석 범위 밖의 "멀티턴 query reformulation" 필요 → 향후 추가 제안 사항으로 문서화

## 4. TYPE 4 — 정답 없음 (거부) 질문

### 4-1. 질문 특성 + 대표 예시
- **특성:** RFP 원문에 **답이 존재하지 않는 정보**를 묻는 질문 (낙찰업체, 납품 완료 여부, 운영 URL, 투입 인원 등). 모델이 "정보 없음"으로 거부하는 것이 정답
- **대표 예시:**
  - Q36: "고려대학교 차세대 포털 시스템 사업의 **최종 낙찰 업체**가 어디야?" (RFP 단계 문서 → 낙찰 정보 없음)
  - Q41: "**광주과학기술원 학사시스템 사업**을 수주한 업체 이름이 뭐야?"
  - Q45: "고려대 포털 시스템을 개발하는 데 **실제로 몇 명이 투입**됐어?"

### 4-2. 3종 검색 결과 동작 패턴
- **수치:** Dense-Sparse 일치율 **70%**, Hybrid-Dense 80%, Hybrid-**Sparse 90%**
- **관찰:** 질문에 기관명·사업명이 명시된 경우가 많아 Sparse의 키워드 매칭이 강하게 작동

**주목 케이스:**

| 질문 | Dense top-1 | Sparse top-1 | Hybrid top-1 | 해석 |
|---|---|---|---|---|
| Q37 (국민연금 납품 완료?) | 한국철도공사 (오매칭) | 국민연금공단 | 국민연금공단 | Dense가 "납품"이라는 일반 단어에 드리프트 |
| Q41 (광주과학기술원 수주업체) | 재단법인 **광주연구원** | 광주과학기술원 | 광주과학기술원 | Dense가 "광주"+"연구" 의미로 엉뚱한 기관에 매칭 |
| Q39 (한국가스공사 컨소시엄) | 한국가스공사 | 한국연구재단 (오매칭) | 한국가스공사 | Sparse가 "한국"+"연구" 매칭 실패, Dense는 정확 |
| Q40 (봉화군 운영 성과) | 경상북도 봉화군 | 경상북도 봉화군 | 경상북도 봉화군 | 3종 모두 정확 (질문에 기관명 명시) |

### 4-3. 원인 진단
**(a) 알고리즘 관점**
- 거부 질문이지만 **검색 단계**에서는 여전히 "관련 문서"를 찾아야 함 (LLM이 그 문서를 보고 "정보 없음"이라 답해야 정답)
- Sparse가 기관명 키워드를 정확히 매칭 → 올바른 문서(해당 RFP)를 가져옴 → LLM이 올바르게 거부 가능
- Dense는 의미적 근접성으로 판단 → "광주" + "연구" 같은 단어 조합에 엉뚱한 문서로 드리프트 가능

**(b) 데이터 관점**
- 질문에 기관명 + 사업명이 거의 항상 명시됨 (TYPE 1과 유사한 패턴)
- 단, 질문 후반부에 "낙찰", "납품", "투입 인원" 등 **RFP에 없는 일반 키워드**가 붙음 → Dense가 이 키워드의 의미에 끌려감

### 4-4. 권장사항 + 한계
- **권장: Sparse 또는 Hybrid** (기관명 기반 안전한 매칭)
- 단, 거부 질문의 최종 성패는 **LLM의 거부 능력**에 달림. 검색이 올바른 문서를 주더라도, LLM이 "정보 없음"을 판단하지 못하면 환각 발생
- **한계:** Retrieval 관점에서 Sparse가 우수하지만, 후단 LLM 프롬프트에서도 "문서에 없는 정보는 답하지 않는다" 제약 필수

### 4-5. Part 04-05 개선 아이디어 연결
- **아이디어 1 (Hybrid RRF)**가 Sparse 강점을 유지하면서 Dense의 오매칭(Q41 광주연구원 같은 케이스)을 보완함
- 이 TYPE은 **검색 개선보다 LLM 프롬프트 개선**이 더 중요 — 본 분석(Part 04) 범위 밖이지만 팀 전체 파이프라인에서 보완 필요

## 6. 결론

1. **단일 방식은 없다.** Dense와 Sparse는 상반된 강점을 가지며, TYPE별로 성패가 갈린다.
2. **Hybrid(RRF)는 안정적 타협.** Dense-Sparse 가중치 0.7:0.3 구성에서, 질문 유형에 따라 자연스럽게 유리한 쪽을 따라가는 패턴이 확인됨.
3. **가장 시급한 개선은 BM25 편향 보정** (TYPE 2 문제). 고려대학교 458청크 outlier로 Sparse의 전체 신뢰도가 하락. → 개선 아이디어 2번에서 직접 처리.
4. **TYPE 3·4의 한계는 Retrieval 계층 단독으로 해결 불가** — 멀티턴 query rewriting, LLM 프롬프트 제약 등 **상위 파이프라인 계층**의 보완이 필요.
