## Type 1 수정사항 (하은 검수): 4문항

### Q05. 질문 변경 — 답할 수 없는 문항 수정
- **문제**: 원본 질문 "입찰 참여 마감일이 언제야?"인데, 문서에 입찰 마감일이 존재하지 않음. Type 1(사실 추출)인데 답할 수 없는 상태
- **변경내용**:
  → 질문을 "공고일은 언제야?"로 변경
  → 메타데이터에 공고일(2024-08-08)이 있어 정상 평가 가능
  → Type 1(12개)/Type 4(10개) 균형 유지 목적

### Q06. hint 구체화 — 채점 기준 부재 해소
- **문제**: 원본 hint "연계 시스템 목록 (PDF 표에서 추출)"은 채점 기준으로 쓸 수 없을 만큼 모호
- **변경내용**:
  → 파싱된 문서에서 확인한 구체적 시스템명 직접 나열
  → 내손안에서울/서울사랑, 서울시 외국어 누리집, Visit Seoul, 서울의 공원 누리집, SDW(서울데이터웨어하우스), 열린데이터광장, S-DoT, 스마트불편신고 등

### Q08. hint 구체화 — 모호한 기준 확정
- **문제**: 원본 hint "예산 1억 5천만 원, 공개일자 확인 필요"에서 "확인 필요"는 채점 시 기준이 없음
- **변경내용**:
  → 메타데이터에서 공개일 2024-04-18 확인하여 명시
  → hint: "예산 1억 5천만 원 (150,000,000원, VAT 포함), 공개일 2024년 4월 18일 (메타데이터 기준)"

### Q10. 예산 오차 수정 + 마감일 누락 보완
- **문제**: 원본 hint "예산 약 1억 5천만 원"인데 실제 문서 금액은 157,300,000원(1억 5,730만 원)으로 730만 원 차이. 질문이 예산+마감일 둘 다 물어보는데 hint에 마감일 정보 누락
- **변경내용**:
  → 정확한 금액 반영: "약 1억 5,730만 원 (157,300,000원, VAT 포함)"
  → 마감일 보완: "입찰 마감일은 문서 본문에 명시 없음 (공고일 메타데이터: 2024-12-09)"

## Type 2 수정사항 (윤민 검수): 5문항

### Q14. 도메인 기준 명확화
- **문제**: "교육·학습 관련 시스템"의 범위가 모호하여 필터링 기준이 불명확
- **변경내용**:
  → 교육·학습(대학 학사시스템, 이러닝, 교육 플랫폼 등)으로 구체화

### Q16. 평가 기준 명시
- **문제**: 기존 ground truth hint가 금액까지 정확히 있어 정답지에 가까움. 내림차순 정렬 여부 평가가 어려움
- **변경내용**:
  → 10억 이상 대형 사업 중심으로 범위 확장
  → "예산 내림차순 정렬" 조건(eval_focus) 명시하여 단순 검색이 아닌 정렬 처리 능력까지 평가 가능하도록 개선

### Q19. 검색 기준 명확화
- **문제**: 특정 사례(챗봇, 영상분석 등)를 ground truth hint에 명시하여 모델이 일부 사례만 답변해도 정답처럼 보이는 bias 발생 (실제로 더 많은 사업을 찾았음에도 오답 처리될 가능성 존재)
- **변경내용**:
  → "AI/인공지능 기술을 명시적으로 요구하는 사업"으로 수정
  → 힌트는 "AI 요구사항 포함 사업 (챗봇, 영상분석, 자동화 등)" 수준으로 일반화 (precision + recall 균형 맞춤)

### Q20. 시간 조건 명확화
- **문제**: 기존 문제 설정인 "2025년 이후"에 부합하는 문서 없음
- **변경내용**:
  → "2024년 이후 입찰 마감"으로 설정
  → hint에 구체 사업명 추가 (한국전기안전공사, GIS 기반 재난안전데이터, 강릉어선안전조업 등)

### Q21. 발주기관 범위 명확화
- **문제**: "인천광역시 관련 기관" 표현이 모호 (지역 vs 기관 혼재)
- **변경내용**:
  → "인천광역시 또는 인천광역시 산하기관"으로 구체화
  → eval_focus에 "유사 기관명 매칭 (인천광역시 vs 산하기관)" 추가

## Type 3 수정사항 (윤민 검수): 5문항

### Q25. 도메인 기준 명확화 (Type 2 Q14와 동일 방식)
- **문제**: Type 2 Q14와 동일하게 "교육·학습" 범위가 모호
- **변경내용**:
  → "교육·학습(이러닝, 학사시스템 등) 관련 사업을 발주한 다른 기관이 있는지 찾아줘"로 구체화

### Q27. 문맥 의존성 강화
- **문제**: "그럼" 표현은 이전 턴과의 연결은 있으나 기준이 불명확
- **변경내용**:
  → "그럼 해당 사업에서 모니터링 업무와 관련된 요구사항"으로 명확화 (동일 문서 내 맥락 유지 및 후속 질문의 명확성 확보)

### Q29. 평가 기준 명확화
- **문제**: "응답 시간" 단일 기준은 누락 가능성 존재
- **변경내용**:
  → "응답 시간 또는 성능(처리속도)"로 확장 (다양한 표현의 성능 요구사항 포착 가능)

### Q30. 평가 기준 명확화
- **문제**: 기존 비교 질문은 판단 기준이 모호
- **변경내용**:
  → "문서 근거를 바탕으로 비교하고, 어느 사업이 상대적으로 더 넓은 범위를 포함하는지 설명해줘"로 구체화 (근거 기반 비교 조건 추가)

### Q33. 검색 기준 명확화
- **문제**: "ERP 관련 사업"은 범위가 넓어 과잉 검색 가능
- **변경내용**:
  → "사업명에 'ERP'가 명시된 구축 사업"으로 제한 (키워드 기반 필터링 precision 향상 및 평가 일관성 확보)

## Type 4 수정사항 (하은 검수): 4문항

Type 4에서 공통적으로 발견된 문제: **문서에 "규칙/요건/계획"은 있는데 "실제 결과"를 물어보는 문항**에서 RAG가 규칙을 실제로 혼동할 위험이 있음. Q39·Q42·Q45 세 문항에 동일한 방식으로 "규칙 vs 실제" 구분 hint를 추가함.

### Q39. hint 추가 — 규칙 vs 실제 혼동 방지
- **문제**: 문서에 공동수급 참여 "규칙"(공동수급 형태로 제안할 경우 등)은 있으나 실제 컨소시엄 구성은 없음. RAG가 규칙을 실제 구성으로 답할 위험
- **변경내용**:
  → hint 추가: "RFP에는 공동수급 참여 규칙/요건만 있음. 실제 참여 컨소시엄 구성은 RFP에 존재하지 않음. 공동수급 규칙을 실제 컨소시엄 구성으로 답하면 오답"

### Q42. hint 추가 — 계획 vs 실제 혼동 방지
- **문제**: 문서에 "계약일~2024.12.31" 계획 기간이 있어서 RAG가 이걸 실제 완료일로 답할 위험
- **변경내용**:
  → hint 추가: "RFP에는 계획 기간만 있음. 실제 완료일은 RFP에 존재하지 않음. 계획된 사업기간을 실제 완료일로 답하면 오답"

### Q43. 질문 변경 — 거부 문항 부적절 해소 (가장 중요한 수정)
- **문제**: 원본 질문 "현재 운영 중인 URL 주소가 뭐야?"인데, 문서에 smap.seoul.go.kr URL이 **실제로 존재**. 거부 문항인데 답할 수 있어서 Type 4로 부적절
- **변경내용**:
  → 질문을 "고도화 이후 신규 서비스 URL이 뭐야?"로 변경
  → 기존 URL은 문서에 있지만, 고도화 이후 신규 URL은 RFP에 없으므로 거부가 타당

### Q45. hint 추가 — 요구사항 vs 실제 혼동 방지
- **문제**: 문서에 투입인력 자격요건, 인력 투입 기준, 참여인력 양식 등 "요구사항"이 많아서 RAG가 요구사항을 실제 투입 인력으로 답할 위험
- **변경내용**:
  → hint 추가: "RFP에는 투입인력 요구사항만 있음. 실제 투입된 인력 수는 RFP에 존재하지 않음. 인력 요구사항을 실제 투입 인력으로 답하면 오답"

## 검수 방법

- **Type 1 (하은)**: 파싱된 문서(data/parsed/*.json)를 열어 각 문항의 hint가 실제 문서 내용과 일치하는지 대조. "hint가 맞는지" 확인
- **Type 2,3 (윤민)**: 질문의 모호성, 검색 기준 명확성, 평가 일관성 관점에서 검수. 질문 리워딩 + eval_focus 강화
- **Type 4 (하은)**: 파싱된 문서에서 해당 정보가 "진짜 없는지" 확인. 유사 정보(규칙/계획)로 인한 RAG 혼동 위험 식별 및 hint 보강
